Wardley Mapping

Strategy as Visual Map

Simon Wardley · 2005 Fotango · "On being lost" (Medium) · 3 鏡頭(Mapping / Doctrine / Climate)

把策略畫成「用戶可見性(Y 軸)× 演化階段(X 軸)」的地圖。看到自己跟競爭者落在演化光譜的哪裡,決定該創新 / 標準化 / 外包還是淘汰。誤判演化階段是組織燒錢主因。

為什麼三鏡頭

Without a map, you're navigating by gut. With a map, you can argue about where you are. — Simon Wardley(改寫自多次 X / Medium 表述)

Wardley 的三層架構不是步驟,是三個鏡頭看同一張策略圖。從哪個鏡頭看由你的當下任務決定:不知道現況 → 用 Mapping;不知道組織該守什麼 → 用 Doctrine;不知道未來會怎麼變 → 用 Climate。三個都熟才算完整策略觀。

Wardley 的學習曲線陡(2014 才公開系統化,2024 才出第一本書 Wardley Maps)。本頁是入口導覽,不是替代品 — 認真用要讀原書 + 自己畫過 10+ 張圖才會有 fluency。

選一個鏡頭(三選一)

Mapping · 地圖鏡頭

把你的服務 / 產品 / 個人工作鏈拆成 components,放到「用戶可見性 × 演化階段」的 2D 地圖上。地圖不解問題 — 它讓你跟同事 / 競爭者爭論「我們究竟在哪」,而不是憑感覺。

下圖是個人品牌寫作者的價值鏈示意:從 Reader(最頂層用戶可見)往下到 Hosting infrastructure(底層 commodity),沿橫軸看每個 component 演化階段。橙色箭頭 = 該 component 預期向右(commodity 方向)移動。

用戶可見性
(Value Chain Visibility)
User Service Component Infra
演化階段 → Evolution(Genesis → Commodity)
Reader讀者(感受)
Quality writing優質寫作
Audience tools受眾經營工具
Hosting主機/CDN

怎麼讀這張地圖

  • 左上(Genesis × User):創新空間。寫作的「讀者真正需要什麼」永遠在這裡 — 你不能用既有公式直接交付,要原創。
  • 右上(Commodity × User):標準化體驗。如 email newsletter 服務 — 對讀者直接,但實現方式已是商品。
  • 左下(Genesis × Infra):基礎研發。如「全新類型的內容呈現方式」,投入大但很可能失敗。
  • 右下(Commodity × Infra):標準基礎設施。如 AWS / Cloudflare — 不該自建,該外包。
  • 橙色箭頭:該 component 必然往 commodity 方向移動。差別只在多快。寫作領域的 AI 工具正在快速從 Custom-built 進 Product 階段,個人寫作者要決定:跟著用 / 抗拒用 / 創造下一代。

練習:畫你自己的圖。先列 5-7 個 components(你的工作流程或商業模式的關鍵元件),為每個決定「用戶感受到嗎?」位置 Y,「跟競爭者一樣嗎?」位置 X。畫完跟一個同行討論,差異處就是學習機會。Atlas v0.4+ 可能會做互動式繪圖 widget。

Doctrine · 原則鏡頭

Wardley 列了 40 條「不管你做哪行都該守的組織原則」,稱為 doctrine。本頁挑 12 條最常用 + 對個人品牌 / NGO 軌道相關性最高的,分 3 組。

溝通 · Communication

沒有共同地圖,所有討論都是各說各話。

D-01

Use a common language

建立共同語言
同一個 component 不同部門叫不同名 → 開會時各說各的。先統一名稱再討論策略。
NGO「使用者 / 服務對象 / 受益人」混用,先選一個再對話
D-02

Be transparent

透明
畫好的地圖要分享 — 不公開的地圖只服務權力,公開的地圖才能被挑戰跟修正。
品牌把你的學習路徑公開寫出來,讀者會幫你找到盲點
D-03

Challenge assumptions

挑戰假設
每個 component 的位置都基於假設。定期問:這假設還對嗎?演化階段是否已經變了?

開發 · Development

同一個團隊不該同時做 Genesis 跟 Commodity。

D-04

Use appropriate methods

用對方法
Genesis 階段用 agile / 探索;Custom-built 用 lean;Product 用 six sigma;Commodity 用 ITIL。錯配方法是常見失敗。
品牌寫新題材用 agile(快發、看反應、調整);成熟題材像「年度回顧」可標準化
D-05

Think small (as in teams)

小團隊
大團隊處理不了 Genesis 級別不確定性。Amazon 兩個披薩規則來自 Wardley。
D-06

Distribute power & decision-making

分散決策權
中央集權適合 Commodity 階段的高度標準化執行,不適合 Genesis 的快速試錯。
NGO新計劃 (Genesis) 給執行小組決策權,例行運作 (Commodity) 才回中央
D-07

Focus on user needs

聚焦使用者需求
地圖的最頂層永遠是 user。任何 component 都要能 trace 到「這如何服務 user」否則該刪。

運營 · Operations

演化階段決定你該自建還是該外包。

D-08

Bias towards action

傾向行動
地圖是輔助決策不是替代決策。畫完要動手,不要陷在 mapping 上完美主義。
D-09

Manage inertia

管理慣性
人會抗拒 component 的演化移動(因為威脅既有技能 / 地位 / 投資)。識別、命名、處理慣性,而非假裝它不存在。
品牌「我的舊文章風格」就是 inertia — 演化階段變了你的風格也該變
D-10

Outsource the commodity

外包 commodity
已演化到 Commodity 的 component 該外包(用 SaaS / 外部服務)。自建 = 燒錢做別人已標準化的事。
NGO會計 / HR / 網站不該自建,該用 SaaS,把資源留給只能自己做的部分
D-11

Optimize flow

優化價值流
看整條鏈,找瓶頸點。優化非瓶頸是浪費。
D-12

Strategy is iterative

策略是迭代
每張地圖都是當下版本,環境變了就重畫。把策略當靜態文件就死了。

Climate · 氣候鏡頭

Climate 是「不管你做什麼都會發生的趨勢」 — 像天氣,你可以準備,不能改變。Wardley 列了 16+ 條,本頁選 8 條最重要的。

C-01

Everything evolves

一切都會演化
所有 component 都會從 Genesis 往 Commodity 移動 — 不會停。差別只在多快。今天的差異化明天是基本款。
品牌個人寫作風格 5 年前的差異化,今天可能已被 AI 工具 commodity 化
C-02

Inertia exists at every stage transition

每個階段轉換都有慣性
Custom-built → Product 時投資的人會抗拒;Product → Commodity 時運營的人會抗拒。慣性可預測,可對沖。
C-03

Co-evolution of practice

實踐共同演化
component 演化會帶動相關 practice 演化。雲端 commodity 化 → DevOps 實踐成形。AI commodity 化 → 寫作實踐正在重塑。
品牌AI 寫作工具的 commodity 化正在催生新的「人寫 + AI 校」分工實踐
C-04

Red Queen effect

紅皇后效應
你必須持續演化才能維持相對位置 — 因為所有人都在演化。不變 ≠ 穩定,等於相對退步。
C-05

Capital flows to genesis

資金流向 Genesis
VC / 創新預算永遠追 Genesis 階段,因為差異化空間最大。商品化階段資金不來 — 那是現金牛分配的地方。
C-06

No choice on evolution

演化無選擇
你不能決定 component 該不該演化 — 只能決定你的位置(領先 / 跟隨 / 棄守)。「我不要它演化」不是選項。
C-07

Componentization enables novelty

元件化使新事物可能
下層 component commodity 化 → 釋放資源做上層 Genesis。電力標準化才有電冰箱;雲端標準化才有 SaaS。
NGOHR / 會計外包到 SaaS 釋放的人力,可投入真正的 Genesis 工作
C-08

Past success breeds inertia

過去的成功養出慣性
越成功的組織越難演化 — 既有模式贏過所以強化,環境變了就重摔。Kodak / Nokia / BlackBerry 都是這個故事。

協同範式

Cynefin

空間(Wardley)+ 問題本質(Cynefin)= 完整策略觀。同一個 component 在 Wardley 上的位置,跟它在 Cynefin 哪個域,是正交軸 — Genesis × Complex 是純探索,Commodity × Clear 是全力自動化。

Effectuation

Wardley 看現況、Effectuation 從現況出發找行動。先 map 再 act,避免「該做但不知怎開始」的癱瘓。

Meadows 12 槓桿點

Doctrine 鏡頭跟 Meadows 槓桿點是同一個思維家族 — 都在問「動哪一點最有效?」Wardley 偏向空間定位,Meadows 偏向因果強度。

反模式

  • 學習曲線陡 — Wardley 不是隨手畫的。沒讀過原書直接畫的圖通常 component 位置全錯,但作者不自知,反而強化錯誤判斷。建議讀 Simon 的 Medium 文章後再開始。
  • 需要時間蒸餾 — Wardley 適合戰略決策(年度 / 重大轉型),不適合每日決策。日常用 OODA / Cynefin。
  • 把地圖當答案 — 地圖只是討論基礎,不是答案。畫完地圖以為「策略做完了」是常見錯誤。地圖是讓你跟同事爭論「我們究竟在哪 + 該往哪去」的工具。
  • 把演化階段當成熟度評分 — Genesis 不是「不成熟」,Commodity 不是「優秀」。它們只是時間軸位置。Genesis 階段的 component 對組織價值不一定低於 Commodity 階段的 component。

出處與校準

Wardley Mapping 由 Simon Wardley 2005 年在英國雲端服務商 Fotango 任 CEO 時發展,2007 起在 LEF(Leading Edge Forum)系統化,2014 公開化(Medium 系列文章),2024 正式出版書籍 Wardley Maps。本頁的 Mapping / Doctrine / Climate 三鏡頭分法來自 Simon 的 LEF 訓練教材 + 個人 Medium 講解。

40 條 Doctrine 完整清單見 Simon Wardley Medium 專欄;本頁挑 12 條時偏向個人品牌寫作 / NGO governance 軌道相關性,略過了純企業 IT 場景的(例如 "Strategy is complex" / "Pivot")。Climate 16+ 條同樣有取捨。互動式地圖繪製工具見 OnlineWardleyMaps(Damon Skelton 開發,非 Simon 官方)。

本頁示意地圖(個人品牌寫作者價值鏈)是作者改寫範例,非 Simon 原版示範。Simon 原版示範多為英國政府 / 雲端服務 / 媒體業 case study。
